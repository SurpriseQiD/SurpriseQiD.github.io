    LLM工具调用、规划与记忆：可执行智能体的三大支柱

    1 三支柱视角

    一个可落地的LLM Agent可以抽象为：

    $$\text{Agent} = f(\text{Planner}, \text{Tools}, \text{Memory})$$

    • Planner决定“做什么”
    • Tools决定“怎么做”
    • Memory决定“基于什么经验做”

    缺一不可。没有工具，Agent无法闭环；没有规划，Agent会短视；没有记忆，Agent无法积累能力。

    2 工具调用设计

    2.1 选择策略

    工具选择可建模为分类问题：

    $$a_t = \arg\max_{a \in \mathcal{A}} P(a|s_t)$$

    其中 $\mathcal{A}$ 包含“直接回答”与各类工具。实践中可加入规则先验（高风险场景必须调用校验工具）。

    2.2 错误恢复

    建议定义统一错误码：

    1. 可重试错误（网络超时）
    2. 参数错误（Schema不匹配）
    3. 业务拒绝（权限不足）

    Agent应具备“自动修参 + 重试 + 降级”链路。

    3 规划范式对比

    1. 一步决策：低延迟，适合简单问答
    2. 线性计划：适合流程型任务
    3. 树搜索计划：适合探索型复杂任务

    在复杂任务中可采用“宽度受限树搜索”：

    $$\pi^* = \arg\max_{\pi \in \Pi_B} \sum_{t=1}^{T} \gamma^{t-1} r_t$$

    其中 $\Pi_B$ 是受分支系数 $B$ 约束的计划集合。

    4 记忆系统

    4.1 短期记忆压缩

    长对话直接拼接会导致上下文污染，应定期摘要：

    1. 保留目标、约束、最新状态
    2. 删除冗余讨论与无效尝试
    3. 记录关键决策理由

    4.2 长期记忆检索

    长期记忆建议采用向量检索 + 元数据过滤，检索评分可写为：

    $$\text{score}(d,q)=\alpha\cdot\cos(e_d,e_q)+\beta\cdot\text{recency}(d)+\eta\cdot\text{trust}(d)$$

    5 与RAG的关系

    RAG解决“知识补给”，记忆系统解决“经验沉淀”。两者配合：

    • RAG提供客观事实
    • Memory提供历史偏好与执行经验

    在企业场景中，建议将“政策规则”放RAG，“用户偏好与失败案例”放Memory。

    6 评估框架

    建议建立四维评估：

    1. 准确性：结果是否正确
    2. 可执行性：调用链是否闭环
    3. 经济性：token/API成本
    4. 稳定性：失败后恢复能力

    可定义综合分数：

    $$S = w_1A + w_2E - w_3C + w_4R$$

    7 工程清单

    1. 工具Schema统一与版本化
    2. 计划器输出结构化中间态
    3. 记忆写入阈值与垃圾回收
    4. 全链路日志与离线回放
    5. 基准任务集持续回归

    8 小结

    工具调用、规划与记忆是Agent工程的“铁三角”。真正可靠的系统，不是偶尔表现好，而是在高压、异常、长流程条件下依然稳定完成任务。
